## Summary
Implement email notification support via SMTP to enable automated communications for group invites, account activities, and other user notifications. The implementation should follow the existing layered architecture and configuration patterns.

## Problem
Currently, the API does not have email notification capabilities. Users cannot receive automated emails for invites, welcome messages, or other important notifications. This requires adding:
1. Email service abstraction and SMTP implementation
2. Configuration management for SMTP credentials
3. Integration with existing services (GroupService, AuthService, SettlementService)
4. Email template support for various notification types

## Scope

### 1. Configuration Infrastructure
- Add `EmailOptions.cs` to `src/MambaSplit.Api/Configuration/` with properties:
  - `SmtpServer` (hostname)
  - `SmtpPort` (default: 587)
  - `SmtpUsername` (authentication username)
  - `SmtpPassword` (authentication password)
  - `FromEmail` (sender address)
  - `FromDisplayName` (sender display name)
  - `EnableSSL` (default: true)
- Add configuration binding in `Program.cs`
- Update `appsettings.json`, `appsettings.Development.json`, and `appsettings.local.example.json`

### 2. Email Service Implementation
- Create `IEmailService` interface in `src/MambaSplit.Api/Services/` with methods:
  - `SendEmailAsync(recipientEmail, recipientName, subject, htmlBody, cancellationToken)`
  - `SendInviteEmailAsync(recipientEmail, inviteToken, groupName, cancellationToken)`
  - (Optional) `SendWelcomeEmailAsync(userEmail, displayName, cancellationToken)`
  - (Optional) `SendSettlementConfirmationAsync(emails, settlementDetails, cancellationToken)`

- Create `EmailService` implementation using `System.Net.Mail.SmtpClient`
  - Proper error handling and logging
  - Async/await pattern throughout
  - Graceful fallback behavior if email sending fails (log but don't crash)

### 3. Service Layer Integration
- Update `GroupService.CreateInviteAsync()` to call `_emailService.SendInviteEmailAsync()` when invite is created
- Update `AuthService.SignupAsync()` to send welcome email (optional)
- Update `SettlementService.CreateSettlementAsync()` to send settlement confirmation emails (optional)
- Ensure email failures do not block core business logic (use try-catch with logging)

### 4. Dependency Injection
- Register `IEmailService` as `Scoped` in `Program.cs`
- Inject into services that need it

### 5. Email Templates (Optional Phase 1)
- Create `src/MambaSplit.Api/Templates/` folder
- Add HTML email templates:
  - `InviteEmail.html` — Group invite with accept link
  - `WelcomeEmail.html` — Welcome message for new users
  - `SettlementConfirmation.html` — Settlement recorded notification

### 6. Configuration Keys
Add to appsettings configuration hierarchy:
```
app:email:smtpServer
app:email:smtpPort
app:email:smtpUsername
app:email:smtpPassword
app:email:fromEmail
app:email:fromDisplayName
app:email:enableSSL
```

## Acceptance Criteria

1. **Configuration**
   - Email settings can be configured via `appsettings.json`, environment variables, or local overrides
   - Default SMTP port is 587, SSL/TLS is enabled by default
   - Configuration is optional (service gracefully handles missing configuration)

2. **Email Service**
   - `EmailService` implements `IEmailService` interface
   - Sends emails via configured SMTP server
   - Handles network/SMTP errors gracefully (log but don't throw)
   - Support async cancellation tokens throughout

3. **Group Invite Emails**
   - When a new invite is created via `GroupService.CreateInviteAsync()`, an email is sent to recipient
   - Email contains group name, sender name, and accept/decline options
   - Email sending failure does not prevent invite creation (logged as warning)

4. **Integration**
   - All services that trigger emails are updated to call email service
   - Email sending is not in critical path (background-friendly)
   - Logging includes email send success/failure for debugging

5. **Testing**
   - Integration tests cover email service with mock SMTP
   - Tests verify email is sent after invite creation
   - Tests verify graceful handling of SMTP failures
   - Existing invite/auth/settlement tests continue to pass

6. **Documentation**
   - README updated with email configuration instructions
   - Example configuration provided in `appsettings.local.example.json`
   - Code comments explain email service pattern

## API Contract Changes
No new endpoints required. Email sending is transparent and triggered by existing operations.

## Non-Breaking
- Email service is purely additive
- Existing API endpoints and logic remain unchanged
- If email is not configured, the API continues to function (emails simply not sent)

## Rollout Notes
1. Implement email service infrastructure first
2. Integrate with group invites (highest priority)
3. Add to auth service (welcome emails) in phase 2
4. Add settlement confirmations in phase 3
5. Deploy with email disabled in production initially, enable after testing
