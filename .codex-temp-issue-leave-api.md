## Summary
Add a backend API that allows a member to leave a group, and automatically rebalance all eligible unsettled expenses equally across the remaining active members.

This issue should follow the exact implementation pattern used for issue #17:
- use `EqualSplitCalculator.Compute(...)` for deterministic split generation
- add a `GroupMembershipService` method for membership removal + rebalance
- keep the database transaction owned by the caller in `GroupService`
- rewrite `expense_splits` in place for eligible expenses
- roll back the whole operation if any step fails

## Goals
1. Support removing a member from a group.
2. Rebalance eligible unsettled expenses equally across the remaining active member set.
3. Keep the operation atomic: if rebalance fails, membership removal must roll back.
4. Reuse the existing deterministic split logic in `EqualSplitCalculator.Compute(...)`.

## Business Rules
1. Trigger rebalance only when a membership record is actually removed.
2. Rebalance only eligible unsettled expenses in the group.
3. An expense is considered unsettled when it has no link in `settlement_expenses`.
4. Exclude reversal entries and reversed originals from rebalance eligibility.
5. Preserve the original expense header:
   - same expense id
   - same amount
   - same payer
   - same createdBy
   - same createdAt
6. Only `expense_splits` should change for eligible expenses.
7. If rebalance fails after membership removal starts, the whole operation must roll back.
8. Reject the operation if removing the member would leave zero active members in the group.
9. Reject the operation if the departing member is the group owner for now, unless ownership transfer is explicitly implemented as part of this issue.
10. If the user is not a member of the group, fail cleanly with API behavior consistent with existing patterns.

## Recommended Design
Mirror issue #17 with a new method in `GroupMembershipService`, for example:

- `RemoveMemberAndRebalanceAsync(Guid groupId, Guid userId, CancellationToken ct = default)`

Responsibility split:
- `GroupService`
  - validate caller authorization
  - open the database transaction
  - delegate membership removal + rebalance to `GroupMembershipService`
  - commit on success
- `GroupMembershipService`
  - confirm membership exists
  - remove membership
  - load remaining active member ids
  - rebalance eligible unsettled expenses using `EqualSplitCalculator.Compute(...)`
  - save changes

Important:
- Keep transaction ownership in `GroupService`, matching the actual issue #17 implementation.
- `GroupMembershipService` should assume it is called inside an active transaction.

## API Surface
Add a leave-group endpoint for the authenticated member.

Recommended first pass:
- `DELETE /api/v1/groups/{groupId}/members/me`

Recommended behavior:
- authenticated member can leave their own group
- return `204 No Content` on success
- frontend refreshes group state after success

Out of scope for this issue:
- removing arbitrary other members
- ownership transfer flows
- admin removal flows

## Transaction Requirements
Run the following in a single transaction:
1. Validate membership and authorization
2. Remove membership
3. Query eligible unsettled expenses
4. Delete existing `expense_splits` for eligible expenses
5. Recreate equal splits across remaining active members using `EqualSplitCalculator.Compute(...)`
6. Save and commit

If any step fails, roll back the whole transaction.

## Implementation Spec

### Current implementation pattern from issue #17
Issue #17 currently uses:
- `EqualSplitCalculator.Compute(...)` for deterministic equal-split logic
- `GroupMembershipService.AddMemberAndRebalanceAsync(...)` for membership insert + rebalance
- `GroupService` as the transaction owner
- in-place `expense_splits` rewrite for eligible unsettled expenses

This issue should mirror that pattern closely.

### Data handling strategy
For the first implementation:
- do not create new expense rows
- do not create reversal rows for leave rebalance
- do not modify expense headers
- rewrite `expense_splits` in place for eligible expenses

Per eligible expense:
1. load remaining active member ids after membership removal
2. delete existing `expense_splits` rows for that expense
3. recreate equal split rows using `EqualSplitCalculator.Compute(...)`

### Eligibility rules
Use the same eligibility logic as issue #17:
- expense belongs to the target group
- `ReversalOfExpenseId == null`
- no other expense exists with `ReversalOfExpenseId == expense.Id`
- no row exists in `settlement_expenses` for that expense

### Rebalance algorithm
Reuse the existing shared helper:
- `EqualSplitCalculator.Compute(totalAmountCents, memberIds)`

This preserves deterministic behavior:
- sort member ids by `Guid.ToString()`
- distribute remainder to the first N members

### Failure behavior
If any of the following fail:
- membership delete
- eligible expense query
- expense split delete
- expense split recreate
- save changes

then the entire operation must roll back.

### Edge cases
Support these cases explicitly:
- member leaves a group with no expenses: removal succeeds, no rebalance work
- member leaves a group with only settled expenses: removal succeeds, no rebalance work
- member leaves a group with reversed expenses present: reversed rows and reversed originals are ignored
- attempt to leave when not a member: fail cleanly
- attempt to leave as the last remaining member: reject
- attempt to leave as group owner: reject for now

## Acceptance Criteria
- A group member can leave a group through a new DELETE API.
- Eligible unsettled expenses are rebalanced equally across remaining active members.
- Settled expenses remain unchanged.
- Reversed expenses and reversed originals remain unchanged.
- Group detail responses reflect updated split rows and net balances after member removal.
- If rebalance fails, membership removal is rolled back.
- The leave operation is rejected when it would leave the group with zero active members.
- The leave operation is rejected for owner-members unless ownership transfer support is added.

## Tests
Add integration tests covering:
1. leaving a group with one unsettled expense rebalances splits across remaining members
2. leaving a group with multiple unsettled expenses rebalances all eligible expenses
3. settled expenses are excluded from rebalance
4. reversed expenses and reversed originals are excluded from rebalance
5. leaving a group with no expenses succeeds without rebalance work
6. leaving a group with only settled expenses succeeds without rebalance work
7. if rebalance persistence fails, membership removal rolls back
8. deterministic remainder allocation is preserved after leave rebalance
9. owner cannot leave group under current rules
10. last remaining member cannot leave group
11. non-member leave attempt fails cleanly
12. group details reflect updated balances after leave

Issue #17's current integration tests cover the happy-path rebalance scenarios and eligibility rules, but do not yet cover rollback-on-failure. This issue should add that coverage for the leave flow.
